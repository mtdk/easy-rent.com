# 用户角色调整说明

> 文档状态：历史（归档参考）
> 口径说明：以 docs/主任务看板.md 与 docs/任务清单.md 为准。


## 调整背景

根据最新需求，收租管理系统第一期仅涵盖管理员和房东两种角色，租客功能作为未来扩展项。本调整旨在优化系统设计，移除当前版本的租客相关功能，同时保留扩展接口便于未来扩展。

## 调整内容

### 1. 用户角色定义（调整后）

| 角色 | 权限说明 | 功能范围 |
|------|----------|----------|
| **管理员** | 系统最高权限，管理所有功能 | - 系统设置管理<br>- 用户管理（管理员、房东）<br>- 所有房产查看和管理<br>- 所有合同查看和管理<br>- 财务报表查看<br>- 系统监控和维护 |
| **房东** | 房产所有者，管理自己的房产 | - 个人资料管理<br>- 自有房产管理<br>- 自有合同管理<br>- 租金收取记录<br>- 财务报表查看（仅自己） |
| **租客** | （未来扩展）房产租用者 | - 预留扩展接口<br>- 数据库设计保留字段<br>- 代码结构支持未来扩展 |

### 2. 功能范围调整

#### 移除的功能（当前版本）：
1. **租客注册和登录**：仅支持管理员和房东账号
2. **租客个人中心**：租客相关的个人信息管理
3. **租客自助服务**：在线查看账单、支付租金、提交维修申请等
4. **租客通知**：针对租客的特定通知功能

#### 保留的功能（当前版本）：
1. **租客信息管理**：房东可以管理自己房产的租客信息（作为合同关联方）
2. **租客数据存储**：数据库保留租客相关字段，用于合同关联
3. **扩展接口**：代码层面预留租客功能扩展点

#### 新增的设计考虑：
1. **租客作为合同关联方**：租客信息作为合同的一部分，由房东管理
2. **未来扩展标记**：在代码中添加 `// TODO: 租客功能扩展` 注释
3. **数据库预留字段**：保留必要的租客相关字段，但简化当前功能

### 3. 数据库设计调整

#### 用户表调整：
```sql
-- 调整前：role ENUM('admin', 'landlord', 'tenant')
-- 调整后：role ENUM('admin', 'landlord')，但保留tenant值支持
CREATE TABLE users (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(100) UNIQUE,
    phone VARCHAR(20) UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    full_name VARCHAR(100),
    avatar VARCHAR(255),
    -- 当前仅支持admin和landlord，但保留tenant枚举值用于未来扩展
    role ENUM('admin', 'landlord', 'tenant') DEFAULT 'landlord',
    is_active BOOLEAN DEFAULT TRUE,
    -- 其他字段保持不变
    ...
);
```

#### 合同表调整：
租客信息仍然需要存储，但作为合同的一部分而非独立用户：
```sql
CREATE TABLE contracts (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    property_id INT UNSIGNED NOT NULL,
    landlord_id INT UNSIGNED NOT NULL,
    -- 租客信息直接存储（非用户表关联）
    tenant_name VARCHAR(100) NOT NULL,
    tenant_phone VARCHAR(20),
    tenant_id_card VARCHAR(50), -- 身份证号
    tenant_email VARCHAR(100),
    -- 未来扩展：关联用户表
    tenant_user_id INT UNSIGNED NULL, -- 预留字段，未来关联users表
    ...
);
```

### 4. 权限控制调整

#### 当前权限矩阵：
| 功能模块 | 管理员 | 房东 |
|----------|--------|------|
| 用户管理 | ✓ 全部 | ✗ 无 |
| 房产管理 | ✓ 全部 | ✓ 仅自己 |
| 合同管理 | ✓ 全部 | ✓ 仅自己 |
| 财务管理 | ✓ 全部 | ✓ 仅自己 |
| 系统设置 | ✓ 全部 | ✗ 无 |

#### 权限检查代码调整：
```php
class Auth {
    // 当前支持的角色
    const ROLE_ADMIN = 'admin';
    const ROLE_LANDLORD = 'landlord';
    const ROLE_TENANT = 'tenant'; // 预留，当前不使用
    
    public function canAccessModule($module) {
        $userRole = $this->getUserRole();
        
        // 当前版本仅处理admin和landlord
        if ($userRole === self::ROLE_TENANT) {
            // 租客角色当前不支持，返回false或跳转到未授权页面
            return false;
        }
        
        // 原有的权限检查逻辑
        $permissions = $this->getRolePermissions($userRole);
        return in_array($module, $permissions);
    }
    
    // 预留租客权限检查方法
    public function isTenant() {
        return $this->getUserRole() === self::ROLE_TENANT;
    }
}
```

### 5. 界面调整

#### 登录/注册界面：
- 移除"租客注册"选项
- 登录页面仅显示管理员和房东登录说明
- 注册功能仅对管理员开放（用于创建房东账号）

#### 导航菜单调整：
- 移除租客相关的菜单项
- 房东界面简化，专注于房产和合同管理
- 管理员界面保持不变

#### 租客信息管理：
- 在合同管理页面中集成租客信息输入
- 租客信息作为合同的一部分进行管理
- 提供租客信息查询和导出功能

### 8. 调整影响评估

#### 正面影响：
1. **开发周期缩短**：减少租客相关功能的开发工作量
2. **系统复杂度降低**：简化权限管理和界面设计
3. **部署更简单**：减少初始配置和培训成本
4. **聚焦核心功能**：专注于房东和管理员的核心需求

#### 潜在风险：
1. **未来扩展成本**：需要确保当前设计支持未来扩展
2. **数据迁移**：未来添加租客功能时可能需要数据迁移
3. **用户体验一致性**：确保当前设计不影响未来租客功能的集成
