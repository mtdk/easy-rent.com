# 移动端范围评估与实施方案（裁剪版）

> 文档状态：有效（执行方案）
> 口径说明：以 docs/主任务看板.md 与 docs/任务清单.md 为准。

> 版本：v1.0
> 日期：2026-04-07
> 适用阶段：P2

## 1. 目标与边界

目标：在不引入原生 App 开发成本的前提下，优先提升手机访问可用性，形成“先可用、再优化、可扩展”的移动端路线。

边界：
- 本方案聚焦“移动 Web 适配 + 关键流程体验优化”。
- 不包含 iOS/Android 原生 App 开发。
- 不包含在线支付能力（当前产品范围已冻结）。

## 2. 现状评估

现状结论：
- 现有系统基于 Bootstrap，本身具备基础响应式能力，但关键业务页在手机端存在操作密度高、信息层级深、筛选区拥挤等问题。
- 已完成部分移动友好修正（如对账页快捷提示在小屏隐藏），但尚未形成系统化标准与验收基线。
- 现阶段最优解不是“重做前端框架”，而是“按业务价值分层改造现有页面”。

## 3. 范围定义（裁剪版）

### 3.1 必做范围（M0）

- 全局导航与操作区移动端规范：
  - 顶部按钮区分组、折叠、主次按钮层级统一。
  - 表格页提供移动端简化视图（卡片或关键列保留）。
- 高价值页面优先适配：
  - 支付与账单列表
  - 月度对账页
  - 财务报表页
  - 合同到期提醒页
- 统一交互基线：
  - 触控目标最小尺寸、间距、输入控件高度。
  - 筛选区双行/折叠规则。
- 移动端验收清单初版：
  - 360px、390px、414px 三类宽度通过。

### 3.2 增强范围（M1）

- 关键动作加速：
  - 常用筛选预设
  - 最近使用条件记忆
- 报表移动端摘要卡：
  - 保留核心指标，复杂图表可折叠。
- 弱网场景优化：
  - 页面资源体积控制与首屏优先加载。

### 3.3 暂不纳入（M2 以后评估）

- 原生 App。
- 小程序双端并行。
- 离线数据库同步。

## 4. 实施路线与工期估算

### 阶段 A（1 周）：标准与样板页

输出：
- 移动端 UI 规范（按钮、筛选区、表格、卡片）。
- 2 个样板页改造（支付列表、月度对账）。

估算：
- 前端改造：3 人日
- 联调与回归：2 人日
- 合计：5 人日

### 阶段 B（1 周）：高价值页面铺开

输出：
- 财务报表、合同提醒等页面完成移动端适配。
- 形成统一组件模式（可复用片段）。

估算：
- 页面改造：4 人日
- 联调与修复：2 人日
- 合计：6 人日

### 阶段 C（0.5 周）：验收与发布

输出：
- 移动端验收报告。
- 发布清单与回归记录。

估算：
- 验收与文档：2 人日

总估算：13 人日（约 2.5 周，1 名前端 + 1 名测试并行）。

## 5. 风险与应对

- 风险：旧页面大量内联 HTML，改造一致性难。
  - 应对：先建立“样板页模板”，再批量套用。
- 风险：表格信息过多，移动端可读性差。
  - 应对：定义“关键列优先 + 详情二级展开”策略。
- 风险：回归范围扩大。
  - 应对：沿用现有回归脚本，新增移动端人工清单与截图证据。

## 6. 验收标准

- 关键页面在 360px 宽度可完成主流程操作，无横向滚动阻断。
- 关键动作（筛选、查看详情、收据跳转）在移动端可 3 步内完成。
- 不降低现有桌面端功能可用性。
- 发布前通过现有自动化回归与移动端人工验收清单。

## 7. 决策结论

- 结论：采用“移动 Web 裁剪改造”路线，暂不启动原生 App。
- 状态：本方案评估完成，可进入 M0 开发排期。
- 下一步建议：将“阶段 A 样板页改造”纳入下一轮迭代任务。
