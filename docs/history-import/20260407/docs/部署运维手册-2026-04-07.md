# 部署运维手册（P2）

> 文档日期：2026-04-07
> 适用范围：EasyRent 单机/内网部署与日常运维

## 1. 目标

- 建立统一的部署与巡检流程。
- 降低发布风险并提升故障可恢复性。
- 形成值班可执行的标准化操作手册。

## 2. 关键脚本总览

- 一键回归：scripts/run-regression.sh
- 夜间总控：scripts/run-nightly-maintenance.sh
- 运维巡检总控：scripts/run-ops-health-check.sh
- 备份：scripts/backup-easy-rent.sh
- 恢复总控：scripts/run-restore-and-verify.sh
- 恢复后健康检查：scripts/post-restore-health-check.sh

## 3. 日常巡检流程

### 3.1 预演（值班开始）

- 执行：
  - ./scripts/run-ops-health-check.sh preview
- 期望：
  - 生成 ops-health-check_preview_*.json
  - 无脚本参数错误

### 3.2 正式巡检（建议每日）

- 执行：
  - ./scripts/run-ops-health-check.sh commit http://127.0.0.1:8080 <db_user> <db_password> easy_rent 127.0.0.1 3306 24
- 检查项：
  - 关键 HTTP 端点可达
  - storage/logs 与 storage/backups 可写
  - 夜间总控报告新鲜度（默认 24h）
  - 数据库 SELECT 1 可执行

### 3.3 结果判定

- 通过：fail_count = 0
- 失败：fail_count > 0，需按第 6 章进行处置

## 4. 发布前流程

- 执行回归：
  - ./scripts/run-regression.sh
- 执行夜间总控预演：
  - ./scripts/run-nightly-maintenance.sh preview 30 30 90 storage/logs/api-access-archives
- 执行运维巡检：
  - ./scripts/run-ops-health-check.sh commit http://127.0.0.1:8080 <db_user> <db_password>
- 执行备份：
  - ./scripts/backup-easy-rent.sh <db_user> <db_password> easy_rent 127.0.0.1 3306 storage/backups

## 5. 监控与证据留存

- 巡检日志：storage/logs/ops-health-check_<mode>_<timestamp>.log
- 巡检报告：storage/logs/ops-health-check_<mode>_<timestamp>.json
- 夜间总控日志：storage/logs/nightly-maintenance_<mode>_<timestamp>.log
- 夜间总控报告：storage/logs/nightly-maintenance_<mode>_<timestamp>.json
- 恢复总控报告：storage/logs/restore-verify_<mode>_<timestamp>.json

## 6. 故障处置 SOP

### 6.1 HTTP 异常

- 先执行：
  - ./scripts/post-restore-health-check.sh http://127.0.0.1:8080 <db_user> <db_password>
- 若仍失败：
  - 检查 Web 服务进程与端口
  - 检查 .env 与数据库连接配置

### 6.2 数据库异常

- 验证凭据与连通性：
  - mysql -h127.0.0.1 -P3306 -u<db_user> -p<db_password> easy_rent -e "SELECT 1"
- 必要时走恢复总控：
  - ./scripts/run-restore-and-verify.sh preview <backup_file> <base_url> <db_user> <db_password> easy_rent 127.0.0.1 3306

### 6.3 夜间总控报告过旧

- 立即执行一次预演：
  - ./scripts/run-nightly-maintenance.sh preview 30 30 90 storage/logs/api-access-archives
- 若失败，查看 alerts 与报告中的 failed_stage 字段定位

## 7. 值班建议

- 建议班前执行 preview，班中执行 commit。
- 所有失败告警在 30 分钟内完成首次响应。
- 每次处置完成后记录：问题、根因、修复、回归结果。
